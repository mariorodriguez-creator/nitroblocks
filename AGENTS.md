# AGENTS.md

This project is a website built with Edge Delivery Services (EDS) in Adobe Experience Manager Sites as a Cloud Service. As an agent, follow the instructions in this file to deliver code based on Adobe's standards for fast, easy-to-author, and maintainable web experiences.

When searching for documentation, always search `site:www.aem.live` to get results specific to Edge Delivery Services. Do **not** confuse EDS with the Java/JCR/OSGi-based AEM technology stack, nor with Cloudflare/Akamai/Fastly edge compute products. The canonical documentation lives at <https://www.aem.live/> and a machine-readable index is at <https://www.aem.live/llms.txt>.

## Skills

You have access to a set of skills in .claude/skills. Each skill consists of a SKILL.md file, and other files such as scripts and resources, which are referenced from there.

**YOU ARE REQUIRED TO USE THESE SKILLS TO ACCOMPLISH DEVELOPMENT TASKS. FAILING TO DO SO WILL RESULT IN WASTED TIME AND CYCLES.**

### How Skills Work

Each skill is a directory in `.claude/skills/` with the following structure:

```
.claude/skills/
  └── {skill-name}/
      ├── SKILL.md        # Main instructions (required)
      ├── scripts/        # Optional supporting scripts
      └── resources/      # Optional resources (examples, templates, etc.)
```

The SKILL.md file contains detailed instructions that you must follow exactly as written. Skills are designed to:
- Provide specialized workflows for common tasks
- Ensure consistency with project standards and best practices
- Reduce errors by codifying expert knowledge
- Chain together when tasks require multiple skill applications

### Skill Discovery and Execution Process

Always use the following process:

1. **Discovery**: When a new conversation starts, discover available skills by running `./.agents/discover-skills`. This script will show you all available skills with their names, paths, and descriptions without loading everything into context.

2. **Selection**: Use each skill based on its name and description when it feels appropriate to do so. Think carefully about all the skills available to you and choose the best ones to use. Note that some skills may reference other skills, so you may need to apply more than one skill to get things done.

3. **Execution**: When you need to use a skill:
   - Read the full SKILL.md file
   - Announce you are doing so by saying "Using Skill: {Skill Name}"
   - Follow the skill's instructions exactly as written
   - Read any referenced resources or scripts as needed
   - Complete all steps in the skill before moving to the next task

### Available Skills

Skills will be added to `.claude/skills/` as needed for this project. Check the `.claude/skills/` directory or run `./.agents/discover-skills` for the current list of available skills.

**For ALL development work involving blocks, core scripts, or functionality, you MUST start with the content-driven-development skill.** It will orchestrate other skills as needed throughout the development workflow.

**EDS-specific skills** (for AEM Edge Delivery Services): eds-analytics, eds-documentation, eds-naming, eds-styles, eds-wcag. Retired aem-* skills are archived in `.claude/skills/archive/`.

## Project Overview

This project is based on the <https://github.com/adobe/aem-boilerplate/> project and set up as a new project. You are expected to follow the coding style and practices established in the boilerplate, but add functionality according to the needs of the site currently developed.

The repository provides the basic structure, blocks, and configuration needed to run a complete site with `*.aem.live` as the backend. See [The Anatomy of a Project](https://www.aem.live/developer/anatomy-of-a-project) for a full description of how EDS projects are structured.

### Key Technologies
- Edge Delivery Services for AEM Sites — documentation at <https://www.aem.live/>
- Vanilla JavaScript (ES6+), **no transpiling, no build steps, no frameworks** — the project is entirely buildless
- CSS3 with modern features, **no Tailwind, Sass, PostCSS, or other CSS frameworks/preprocessors**
- HTML5 semantic markup generated by the aem.live backend, decorated by our code
- Node.js tooling (linting only)

### Guiding Philosophy: Buildless and Framework-Free

EDS deliberately avoids build tools, bundlers, and frameworks. This keeps the developer experience simple and the site fast. Do not introduce webpack, Vite, Rollup, React, Vue, Angular, Svelte, or similar tools. The simplest solution is to rely on the growing feature set supported by browsers. See [Best Practices for Developers](https://www.aem.live/docs/dev-collab-and-good-practices#frameworks).

## Content First Principles

Content is the foundation of every AEM website. **Always start with content before writing code.** This is the single most important principle in EDS development.

### Start Development with Content

Before writing a single line of CSS or JavaScript:
1. Create sample content in a Word or Google Doc (or spreadsheet)
2. Ensure the content structure feels natural and intuitive for authors
3. Share it with people who have authoring experience for validation
4. Only then begin developing CSS and JS code

See [Best Practices — Start your development with content](https://www.aem.live/docs/dev-collab-and-good-practices#start-your-development-with-content).

### Use Drafts for Content Changes

The content lifecycle is different from the code lifecycle. When proposing changes to content structure or creating new blocks, copy pages into your `/drafts/{your-name}/` folder. After code changes are merged to `main`, authors can copy or merge draft content to its final location. See [Best Practices — Use drafts](https://www.aem.live/docs/dev-collab-and-good-practices#use-drafts).

### Backwards Compatibility

Code merged to `main` should never break existing published content. New functionality should only activate when new content is previewed and published. This doesn't apply to design changes or bug fixes applied across existing content. See [Best Practices — Backwards compatibility](https://www.aem.live/docs/dev-collab-and-good-practices#backwards-compatibility-of-new-features).

### Use Content for Strings and Literals

Strings displayed to end users that could be translated or changed should be sourced from content — use [Placeholders](https://www.aem.live/docs/placeholders) spreadsheets or other authored content. Never hard-code user-visible text in JavaScript or CSS. See [Best Practices — Use content for strings](https://www.aem.live/docs/dev-collab-and-good-practices#use-content-for-stringsliterals).

### Use Content for Static Resources

Do not commit binaries or large static assets (images, HTML files, SVGs) to the GitHub repo unless they are referenced directly from code (e.g., icons). Content-related assets belong in the content source (SharePoint, Google Drive, etc.). See [Best Practices — Use content for static resources](https://www.aem.live/docs/dev-collab-and-good-practices#use-content-for-static-resources).

**Quick tips for inspecting content:**
- Page HTML: `curl http://localhost:3000/path/to/page`
- Source markdown: `curl http://localhost:3000/path/to/page.md`
- Local test content: Use `--html-folder drafts` flag when starting dev server

**For development workflow:** Use the **content-driven-development** skill for all development tasks.

## Code Style Guidelines

### JavaScript
- Use ES6+ features (arrow functions, destructuring, template literals, etc.)
- Follow Airbnb ESLint rules (already configured) — do NOT change linting configuration without team consensus
- Always include `.js` file extensions in imports
- Use Unix line endings (LF)
- All code runs client-side in the browser; use only features supported by browsers that support dynamic `import()` (all evergreen browsers)
- Block JS is loaded as ESM modules and must `export default function decorate(block) { ... }`
- Do NOT use `!important` in CSS or overly complex selectors — add CSS classes via JS decoration instead
- Do NOT add 3rd party libraries to `head.html`; load them where needed via `loadScript()` in the specific block, or use `IntersectionObserver` for lazy loading when the block scrolls into view
- Minification is unnecessary and discouraged; the buildless architecture keeps files small. See [Best Practices — Minification](https://www.aem.live/docs/dev-collab-and-good-practices#minification)

**For detailed JavaScript guidelines:** Use the **building-blocks** skill which includes comprehensive decoration patterns and best practices.

### CSS
- Mobile-first responsive design
- Standard breakpoints: `600px`, `900px`, `1200px` — all `min-width`, never mix `min-width` and `max-width`. Only deviate with strong justification. See [Best Practices — Breakpoints](https://www.aem.live/docs/dev-collab-and-good-practices#breakpoints)
- All selectors in block CSS must be scoped to the block: `.{blockName} .selector` — no side effects on other blocks or default content. See [Best Practices — CSS selector block isolation](https://www.aem.live/docs/dev-collab-and-good-practices#css-selector-block-isolation)
- Prefix CSS classes and variables private to a block with the block name. Shared/inheritable classes should not be prefixed.
- Leverage ARIA attributes (e.g., `[aria-expanded="true"]`) for styling interactive states instead of inventing custom classes. See [Best Practices — Leverage ARIA](https://www.aem.live/docs/dev-collab-and-good-practices#leverage-aria-attributes-for-styling)
- Avoid `!important` — it is almost never necessary when you control the full CSS context. See [Best Practices — The !important rule](https://www.aem.live/docs/dev-collab-and-good-practices#the-important-rule)
- Follow Stylelint standard configuration (already configured) — do NOT change linting configuration
- Keep CSS selectors simple and readable; prefer adding CSS classes in JS over complex selectors. See [Best Practices — CSS selectors complexity](https://www.aem.live/docs/dev-collab-and-good-practices#css-selectors-complexity)
- Use modern CSS features, but ensure support in evergreen browsers. See [Best Practices — Modern CSS features](https://www.aem.live/docs/dev-collab-and-good-practices#modern-css-features)

**For detailed CSS guidelines:** Use the **building-blocks** skill which includes comprehensive styling patterns and best practices.

### HTML
- Use semantic HTML5 elements
- Ensure accessibility standards (ARIA labels, proper heading hierarchy, alt text on images)
- Follow AEM markup conventions for blocks and sections
- All canonical content is rendered server-side; client JS only decorates for display and accessibility

## Key Concepts

### Content

CMS authored content is a key part of every AEM Website. The content of a page is broken into sections. Sections can have default content (text, 
headings, links, etc.) as well as content in blocks.

The AEM backend delivers clean semantic HTML from authored content. The JavaScript in `scripts.js` augments this markup into a decorated DOM. Understand this as a two-step process:

**For development workflow:** Use the **content-driven-development** skill for all development tasks. This skill ensures you identify or create test content before writing code, following AEM best practices.

**Quick tips:**
- Inspect page structure: `curl http://localhost:3000/path/to/page`
- View source markdown: `curl http://localhost:3000/path/to/page.md`
- Local test content: Use `--html-folder drafts` flag when starting dev server

See [Markup, Sections, Blocks, and Auto Blocking](https://www.aem.live/developer/markup-sections-blocks) for the full reference.

#### Sections

Sections group default content and blocks. Authors create section breaks with `---` (horizontal rules). Sections support **Section Metadata** blocks where the `Style` property adds CSS classes to the containing section element. CSS styling is usually the only developer interaction needed with sections.

#### Default Content

Default content uses standard semantic elements (headings, paragraphs, lists, images, links, emphasis) that authors create naturally in their documents. Leave as much content as default content as possible — it's the most natural way for authors to work.

#### Blocks

Blocks are the re-usable building blocks of AEM. Blocks add styling and functionality to content.  They are the primary mechanism for componentizing code. Each block has an initial content structure it expects, and transforms the HTML using DOM APIs to render a final structure.


**Key principle:** The initial content structure is the contract between authors and developers. Design this structure before writing any code, and be careful when making changes that could break existing pages.

Block markup renders as nested `<div>` tags for rows and columns:
```html
<div class="blockname">
  <div>
    <div>
      <p>Hello, World.</p>
    </div>
  </div>
</div>
```

**For creating or modifying blocks:** Use the **building-blocks** skill which guides you through:
- Content model design (via content-driven-development)
- JavaScript decoration patterns
- CSS styling conventions
- Testing and validation

**Block options**: Authors add variants in parentheses — e.g., `Columns (wide)` produces `<div class="columns wide">`. Comma-separated options become separate classes: `Columns (dark, wide)` → `<div class="columns dark wide">`. Multi-word options use hyphens: `Columns (super wide)` → `<div class="columns super-wide">`.

**For creating or modifying blocks:** Use the **building-blocks** skill. Also consult the [Block Collection](https://www.aem.live/developer/block-collection) and [Block Party](https://www.aem.live/developer/block-party/) for reference implementations.

**Tip:** Use `curl http://localhost:3000/path/to/page` to inspect the HTML delivered by the backend before making assumptions.

### Auto-Blocking

Auto-blocking programmatically creates block DOM structures from default content and metadata without authors explicitly authoring tables. This happens early in decoration before blocks are loaded (in `buildAutoBlocks()` in `scripts.js`).

Common use cases:
- **Template-based**: Pages with a shared `template` metadata value get auto-blocks (e.g., article-header combining title, hero image, author, and date metadata)
- **Link-based**: Links to specific services (e.g., YouTube URLs) are auto-wrapped in embed blocks

This keeps content natural for authors while giving developers organized block code. See [Markup — Auto Blocking](https://www.aem.live/developer/markup-sections-blocks#auto-blocking).

Pages are progressively loaded in three phases to maximize performance. This process begins when `loadPage` from scripts.js is called.
* Eager - load only what is required to get to LCP. This generally includes decorating the overall page content to create sections, blocks, 
buttons, etc. and loading the first section of the page.
* Lazy - load all other page content, including the header and footer.
* Delayed - load things that can be safely loaded later here and incur a performance penalty when loaded earlier

### Indexing

A flexible indexing facility tracks content pages as spreadsheets or JSON, commonly used for page lists, feeds, filters, and sorting. See [Indexing](https://www.aem.live/developer/indexing).

## Web Performance — Keeping it 100

Performance is non-negotiable on EDS projects. The boilerplate starts with a perfect Lighthouse 100 score. Your job is to keep it there. **It is hard to improve a score once it drops, but not hard to maintain 100 with continuous testing.** See [Web Performance — Keeping your Lighthouse Score 100](https://www.aem.live/developer/keeping-it-100).

### Three-Phase Loading (E-L-D)

Pages are progressively loaded in three phases. This architecture is critical to performance.

**Phase E — Eager** (before LCP):
- Body starts hidden (`display:none`) to prevent premature image downloads and CLS
- DOM is "decorated": CSS classes added to icons, buttons, blocks, and sections; auto-blocks created
- Body is shown; first section loaded with priority on the LCP candidate (first image)
- Fonts load asynchronously after LCP candidate is displayed
- **Rule of thumb: keep aggregate payload before LCP below 100kb**
- **No connections to second origins before LCP** — TLS/DNS adds significant delay
- Everything in `styles.css` and `scripts.js` is in this path — keep them small and efficient

**Phase L — Lazy** (after LCP, controlled by project):
- Remaining sections and blocks load (JS + CSS)
- Remaining images load via `loading="lazy"`
- Header and footer load asynchronously as separate blocks
- Keep bulk of payload from the same origin; avoid impacting TBT/TTI/FID

**Phase D — Delayed** (≥3 seconds after LCP, 3rd party):
- Marketing tooling, consent management, analytics, chat modules
- Handled in `delayed.js` — this is a catch-all for scripts causing TBT
- Goal: zero blocking time; use web workers or split into small non-blocking files
- Ideally, once TBT problems are fixed, libraries can move to lazy phase

### Performance Anti-Patterns to Avoid

| Anti-Pattern | Why It's Bad | Reference |
|---|---|---|
| Early hints / h2-push / pre-connect before LCP | Consumes the ~100kb bandwidth budget for mobile, delaying LCP | [Keeping it 100 — Early hints](https://www.aem.live/developer/keeping-it-100#early-hints--h2-push--pre-connect-are-part-of-the-network-budget) |
| Preloading fonts | Counterproductive; fonts must load after LCP using [font fallback](https://www.aem.live/developer/font-fallback) technique | [Keeping it 100 — Fonts](https://www.aem.live/developer/keeping-it-100#fonts) |
| 3rd party scripts in `head.html` | In critical path before LCP; load via `loadScript()` in specific blocks or `delayed.js` | [Best Practices — Loading 3rd party libraries](https://www.aem.live/docs/dev-collab-and-good-practices#loading-3rd-party-libraries) |
| Large JS bundles / build tools | Parsing large files creates blocking time; keep files small and buildless | [Best Practices — Minification](https://www.aem.live/docs/dev-collab-and-good-practices#minification) |
| Client-side rendering of canonical content | All canonical content must be server-rendered; client JS only decorates | [Keeping it 100 — Server vs Client rendering](https://www.aem.live/developer/keeping-it-100#server-sided-vs-client-sided-rendering) |
| Path redirects (e.g., / → /en → /en/home) | Each redirect adds a penalty visible in CWV field data | [Keeping it 100 — Redirects](https://www.aem.live/developer/keeping-it-100#redirects-for-paths-resolution) |
| Frameworks (React, Vue, etc.) in the LCP path | Introduce TBT and bloat for simple layout problems | [Best Practices — Frameworks](https://www.aem.live/docs/dev-collab-and-good-practices#frameworks) |

### LCP Optimization

- The LCP element is typically the "hero" image at the top of the page
- Fewer blocks in the first section = faster LCP
- If the LCP element comes from an async lookup (fragment, JSON, service), the loading sequence must wait for the first block to update the DOM before guessing the LCP candidate
- For dual hero images (desktop + mobile), remove the unused one from the DOM to avoid wasting bandwidth
- LCP can also be a video or long text — understand how the LCP candidate is computed for your case
